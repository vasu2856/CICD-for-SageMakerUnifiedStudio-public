# Requirements Document

## Introduction

The `destroy` command is the inverse of the `deploy` command for the SMUS CI/CD CLI tool. It reads a manifest file, identifies the target stage(s), and deletes all resources that were directly created by `deploy` — including QuickSight dashboards/datasets/data sources, S3 objects at declared target paths, Airflow serverless workflows, and optionally the DataZone project. It respects a strict dependency ordering to ensure safe teardown, collects all user confirmations before any destructive action begins, and is idempotent: resources already absent are logged as warnings rather than errors.

The `destroy` command is distinct from the existing `delete` command. The `delete` command only removes the DataZone project and its environments. The `destroy` command removes all deployed content resources (QuickSight, S3, Airflow workflows, workflow-created resources) and conditionally removes the DataZone project.

The `destroy` command is always synchronous. Each deletion step must complete before the next begins, preserving the strict dependency ordering that makes teardown safe. There is no `--async` mode.

For resources created by Airflow workflow runs (e.g. Glue jobs), destroy uses an operator registry approach: it parses the workflow YAML files declared in `content.workflows`, identifies tasks whose operator type is in the supported registry, extracts the resource name from a well-known static field, and deletes those resources directly via boto3. Resources created by operators not in the registry (e.g. `SageMakerNotebookOperator`, which runs notebooks but creates no persistent named resource itself) are skipped and noted in the output. Resources created dynamically inside notebook code (e.g. Bedrock Agents) remain out of scope.

## Assumptions

- The DataZone project and its connections (including `default.s3_shared`) are reachable at the time `destroy` is invoked. S3 bucket and prefix resolution requires a live DataZone API call to the project's connections; destroy does not attempt to infer S3 paths from the manifest alone.
- The AWS credentials available at invocation time have sufficient permissions to list and delete all resource types declared in the manifest (QuickSight, S3, Airflow Serverless, DataZone, Glue).
- The `Resource_Prefix` configured in `deployment_configuration.quicksight.overrideParameters` is unique enough that a prefix-based scan will not match QuickSight resources belonging to other applications. If the prefix scan returns more resources than declared in the manifest, destroy will detect this as a collision and abort before deleting anything.
- Airflow workflow names generated by `_generate_workflow_name` (format: `{app_name}_{project_name}_{dag_name}`) are unique within the AWS account and region. If more than one workflow matches the reconstructed name, destroy will detect this as a collision and abort before deleting anything.
- S3 `targetDirectory` prefixes are unique to this application. Destroy cannot detect S3 prefix collisions across applications; the operator is responsible for ensuring no two manifests share the same `targetDirectory` on the same S3 connection.

## Glossary

- **Destroy_Command**: The `destroy` CLI subcommand implemented in `src/smus_cicd/commands/destroy.py` and registered in `src/smus_cicd/cli.py`.
- **Manifest**: A YAML file conforming to the `ApplicationManifest` schema that declares the application name, content, and per-stage deployment configuration.
- **Stage**: A named deployment target (e.g. `dev`, `test`, `prod`) defined under `stages` in the manifest.
- **StageConfig**: The parsed representation of a single stage entry, including `project`, `domain`, and `deployment_configuration`.
- **DeploymentConfiguration**: The `deployment_configuration` block within a `StageConfig`, containing `storage`, `git`, `quicksight`, and related sub-sections.
- **Resource_Prefix**: The value of `deployment_configuration.quicksight.overrideParameters.ResourceIdOverrideConfiguration.PrefixForAllResources` for a given stage, with `{stage.name}` resolved to the stage key. Used to enumerate all QuickSight resources deployed by this manifest entry.
- **QuickSight_Dashboard**: A QuickSight dashboard whose ID begins with the `Resource_Prefix` for the stage.
- **QuickSight_Dataset**: A QuickSight dataset whose ID begins with the `Resource_Prefix` for the stage.
- **QuickSight_DataSource**: A QuickSight data source whose ID begins with the `Resource_Prefix` for the stage.
- **S3_Target**: An S3 prefix defined by `targetDirectory` in a `deployment_configuration.storage` or `deployment_configuration.git` entry, resolved against the `connectionName` S3 connection.
- **Airflow_Workflow**: An Airflow serverless workflow whose ARN is reconstructed using `generate_workflow_name` and `find_workflow_arn` from the existing `airflow_serverless` helper.
- **Workflow_Run**: An active or running execution of an Airflow_Workflow.
- **DataZone_Project**: The DataZone project associated with a stage, identified by `project.name` and the domain resolved from `domain` config.
- **Force_Flag**: The `--force` / `-f` CLI option that suppresses all interactive confirmation prompts.
- **Operator_Registry**: A mapping of Airflow operator class names to deletion handlers. Each entry defines which task field contains the resource name and how to delete that resource type. The initial registry contains one entry: `GlueJobOperator` → delete Glue job by `job_name` field.
- **Workflow_YAML**: A YAML file in the format parsed by the SMUS CI/CD workflow engine, containing a top-level workflow key with `dag_id` and `tasks`. Each task has an `operator` field and operator-specific parameters.
- **Workflow_Created_Resource**: A cloud resource whose creation is triggered by an Airflow task at workflow runtime, identified by parsing the Workflow_YAML and matching the task's operator against the Operator_Registry.

## Requirements

### Requirement 1: CLI Interface

**User Story:** As a DevOps engineer, I want a `destroy` CLI subcommand with consistent options, so that I can tear down deployed resources using the same conventions as other commands.

#### Acceptance Criteria

1. THE Destroy_Command SHALL accept a `--manifest` / `-m` option specifying the path to the manifest file, defaulting to `manifest.yaml`.
2. THE Destroy_Command SHALL accept a `--targets` / `-t` option specifying a comma-separated list of stage names to destroy, defaulting to all stages defined in the manifest.
3. THE Destroy_Command SHALL accept a `--force` / `-f` flag that suppresses all interactive confirmation prompts.
4. THE Destroy_Command SHALL accept an `--output` / `-o` option accepting `TEXT` (default) or `JSON` values, controlling output format.
5. WHEN `--targets` specifies a stage name not present in the manifest, THE Destroy_Command SHALL print an error listing the invalid stage names and the available stage names, then exit with a non-zero exit code.
6. WHEN `--output JSON` is specified, THE Destroy_Command SHALL write all structured results to stdout as a single JSON object and SHALL route all log/progress messages to stderr.

---

### Requirement 2: Manifest Parsing and Validation

**User Story:** As a DevOps engineer, I want the destroy command to validate the manifest before taking any action, so that configuration errors are caught early.

#### Acceptance Criteria

1. WHEN the manifest file does not exist or fails schema validation, THE Destroy_Command SHALL print a descriptive error message and exit with a non-zero exit code without performing any destructive actions.
2. THE Destroy_Command SHALL use `ApplicationManifest.from_file` to parse the manifest, reusing all existing validation logic.
3. WHEN a stage's `deployment_configuration` is absent or empty, THE Destroy_Command SHALL skip resource deletion for that stage and emit a warning, then continue to the next stage.

---

### Requirement 3: Pre-Destruction Validation and Confirmation

**User Story:** As a DevOps engineer, I want all validations and confirmations to happen before any resource is deleted, so that I can review the full impact and abort safely.

#### Acceptance Criteria

1. THE Destroy_Command SHALL run a full validation cycle across ALL targeted stages before initiating any destructive action or prompting for confirmation. Resource discovery includes live AWS API calls to resolve S3 connection details, enumerate QuickSight resources by prefix, and check for active Airflow workflow runs.
2. DURING the validation cycle, THE Destroy_Command SHALL collect ALL validation errors and warnings encountered across all stages without aborting early. Each issue SHALL be recorded and associated with the stage and resource where it was found.
3. WHEN the validation cycle completes and one or more validation errors were found (e.g. resource collisions), THE Destroy_Command SHALL print a consolidated report of all errors and warnings, then exit with a non-zero exit code WITHOUT performing any destructive actions.
4. WHEN the validation cycle completes with no errors, THE Destroy_Command SHALL print a full destruction plan listing every resource that will be deleted, grouped by stage and resource type, including resource IDs and S3 prefixes.
5. WHEN `--force` is not set, after printing the destruction plan THE Destroy_Command SHALL prompt the user for a single explicit confirmation before proceeding.
6. WHEN the user declines the confirmation prompt, THE Destroy_Command SHALL exit without performing any destructive actions.
7. WHEN `--force` is set, THE Destroy_Command SHALL skip the confirmation prompt and proceed directly to destruction after a clean validation cycle.
8. WHEN one or more targeted stages contain Airflow_Workflows with active Workflow_Runs AND `--force` is not set, THE Destroy_Command SHALL include the active runs in the destruction plan and ask the user whether to force-stop them or abort, as part of the single confirmation step.
9. WHEN the user chooses to abort due to active Workflow_Runs, THE Destroy_Command SHALL exit with a non-zero exit code without performing any destructive actions.
10. WHEN `--force` is set and active Workflow_Runs exist, THE Destroy_Command SHALL automatically proceed with force-stopping the runs without prompting.
11. DURING the validation cycle, WHEN the QuickSight prefix scan for a stage finds more dashboards, datasets, or data sources matching the `Resource_Prefix` than are declared in `content.quicksight`, THE Destroy_Command SHALL record this as a validation error listing all matched resource IDs and the expected count, and continue validating remaining stages.
12. DURING the validation cycle, WHEN more than one Airflow workflow in the account and region matches the reconstructed workflow name for a given manifest entry, THE Destroy_Command SHALL record this as a validation error listing all matched workflow ARNs, and continue validating remaining stages.

---

### Requirement 4: Destruction Ordering

**User Story:** As a DevOps engineer, I want resources to be deleted in a safe dependency order, so that dependent resources are removed before the resources they depend on.

#### Acceptance Criteria

1. THE Destroy_Command SHALL execute destruction in the following strict order for each targeted stage:
   a. Stop active Workflow_Runs (if confirmed or forced)
   b. Delete Workflow_Created_Resources identified from workflow YAML files (e.g. Glue jobs)
   c. Delete Airflow_Workflows
   d. Delete QuickSight dashboards, then datasets, then data sources
   e. Delete S3 objects at all declared S3_Target paths (storage and git)
   f. Delete the DataZone_Project (only if `project.create: true`)
2. THE Destroy_Command SHALL NOT delete the DataZone_Project before all content resources (QuickSight, S3, Airflow) for that stage have been processed.
3. THE Destroy_Command SHALL NOT begin destruction of any resource until the pre-destruction validation and confirmation phase (Requirement 3) is fully complete.

---

### Requirement 5: Airflow Workflow Deletion

**User Story:** As a DevOps engineer, I want Airflow serverless workflows declared in the manifest to be deleted, so that workflow infrastructure is fully cleaned up.

#### Acceptance Criteria

1. FOR EACH workflow entry in `content.workflows`, THE Destroy_Command SHALL reconstruct the workflow ARN using `generate_workflow_name` and `find_workflow_arn` from the `airflow_serverless` helper.
2. WHEN a workflow ARN cannot be found by its reconstructed name, THE Destroy_Command SHALL log a warning and continue without treating the absence as a fatal error.
3. WHEN a workflow has no active Workflow_Runs, THE Destroy_Command SHALL delete it using the `delete_workflow` helper. This step occurs after all Workflow_Created_Resources for that workflow have been deleted (per Requirement 4 ordering).
4. WHEN a workflow has active Workflow_Runs and the user has confirmed force-stop (or `--force` is set), THE Destroy_Command SHALL stop each active run using `stop_workflow_run` before deleting the workflow.
5. THE Destroy_Command SHALL wait for each workflow deletion to complete before proceeding to the next resource type.

---

### Requirement 6: QuickSight Resource Deletion

**User Story:** As a DevOps engineer, I want QuickSight dashboards, datasets, and data sources deployed by the manifest to be deleted, so that QuickSight resources are fully cleaned up.

#### Acceptance Criteria

1. WHEN `deployment_configuration.quicksight` is present for a stage, THE Destroy_Command SHALL enumerate all QuickSight dashboards, datasets, and data sources whose IDs begin with the `Resource_Prefix` for the stage (resolved by applying the same `{stage.name}` variable substitution as `deploy`). All matched resources are candidates for deletion.
2. THE Destroy_Command SHALL use `list_dashboards`, `list_datasets`, and `list_data_sources` to enumerate all QuickSight resources matching the `Resource_Prefix`, and delete all matches. The operator is responsible for ensuring the `Resource_Prefix` is unique enough to avoid matching resources from other applications.
3. WHEN a QuickSight resource is not found by its reconstructed ID, THE Destroy_Command SHALL log a warning and continue without treating the absence as a fatal error.
4. THE Destroy_Command SHALL delete QuickSight dashboards before datasets, and datasets before data sources, within the QuickSight deletion sub-phase.
5. WHEN `deployment_configuration.quicksight` is absent for a stage, THE Destroy_Command SHALL skip QuickSight deletion for that stage.

---

### Requirement 7: S3 Object Deletion

**User Story:** As a DevOps engineer, I want S3 objects at the target paths declared in the manifest to be deleted, so that deployed artifacts are fully cleaned up.

#### Acceptance Criteria

1. FOR EACH entry in `deployment_configuration.storage`, THE Destroy_Command SHALL resolve the S3 bucket and base prefix by calling `get_project_connections` with the stage's `connectionName`, then delete all S3 objects under the resolved `{connection_base}/{targetDirectory}` prefix.
2. FOR EACH entry in `deployment_configuration.git`, THE Destroy_Command SHALL resolve the S3 bucket and base prefix by calling `get_project_connections` with the entry's `connectionName`, then delete all S3 objects under the resolved `{connection_base}/{targetDirectory}` prefix.
3. WHEN an S3 prefix contains no objects (already empty or never populated), THE Destroy_Command SHALL log a warning and continue without treating the absence as a fatal error.
4. THE Destroy_Command SHALL NOT delete S3 objects outside the `targetDirectory` prefixes declared in the manifest.
5. WHEN `deployment_configuration.storage` and `deployment_configuration.git` are both absent for a stage, THE Destroy_Command SHALL skip S3 deletion for that stage.
6. WHEN two or more manifest entries resolve to overlapping S3 prefixes (e.g. `workflows/` is a subdirectory of `bundle/`), THE Destroy_Command SHALL deduplicate the prefix list before deletion to avoid redundant API calls.

---

### Requirement 8: Conditional DataZone Project Deletion

**User Story:** As a DevOps engineer, I want the DataZone project to be deleted only when the manifest declares that destroy owns it, so that shared projects are never accidentally removed.

#### Acceptance Criteria

1. WHEN `project.create` is `true` for a stage, THE Destroy_Command SHALL delete the DataZone_Project for that stage after all other resources have been deleted.
2. WHEN `project.create` is `false` or absent for a stage, THE Destroy_Command SHALL NOT delete the DataZone_Project for that stage.
3. WHEN `project.create` is `true` and the DataZone_Project is not found, THE Destroy_Command SHALL log a warning and continue without treating the absence as a fatal error.
4. THE Destroy_Command SHALL use the existing `get_domain_from_target_config`, `get_project_id_by_name`, and `delete_project` helpers from `src/smus_cicd/helpers/datazone.py` for project deletion.

---

### Requirement 9: Idempotency and Error Resilience

**User Story:** As a DevOps engineer, I want running destroy multiple times to be safe, so that re-running after a partial failure does not cause errors.

#### Acceptance Criteria

1. WHEN a resource targeted for deletion is not found (already deleted or never created), THE Destroy_Command SHALL log a warning and continue processing remaining resources.
2. THE Destroy_Command SHALL NOT exit with a non-zero exit code solely because a resource was not found during deletion.
3. WHEN a deletion API call returns a non-recoverable error (not a "not found" error), THE Destroy_Command SHALL log the error, record the failure in the results, and continue processing remaining resources in the stage.
4. WHEN one or more resource deletions fail with non-recoverable errors, THE Destroy_Command SHALL exit with a non-zero exit code after all stages have been processed.
5. FOR ALL stages where all targeted resources are either successfully deleted or were already absent, THE Destroy_Command SHALL report a successful status for that stage.

---

### Requirement 10: Output and Reporting

**User Story:** As a DevOps engineer, I want clear output from the destroy command in both human-readable and machine-readable formats, so that I can integrate it into automated pipelines and review results interactively.

#### Acceptance Criteria

1. WHEN `--output TEXT` is set (or defaulted), THE Destroy_Command SHALL print a per-resource status line for each resource processed, using visual indicators to distinguish success, warning (not found), and error states.
2. WHEN `--output JSON` is set, THE Destroy_Command SHALL print a single JSON object to stdout containing the application name, targeted stages, and a per-stage list of resource results including resource type, resource ID, and status (`deleted`, `not_found`, `error`, `skipped`).
3. THE Destroy_Command SHALL print a summary at the end of execution showing the count of resources deleted, not found, skipped, and errored per stage.
4. WHEN `--output JSON` is set, THE Destroy_Command SHALL NOT print any non-JSON content to stdout; all progress and warning messages SHALL be written to stderr.

---

### Requirement 11: Scope Boundary — Workflow-Created Resources

**User Story:** As a DevOps engineer, I want the destroy command to clean up resources created by workflow tasks where possible, and clearly communicate what it cannot clean up, so I know what requires manual intervention.

#### Acceptance Criteria

1. THE Destroy_Command SHALL delete Workflow_Created_Resources for all operator types present in the Operator_Registry, by parsing the Workflow_YAML files for each workflow declared in `content.workflows`.
2. WHEN a task's operator type is not in the Operator_Registry, THE Destroy_Command SHALL skip that task, log a message identifying the operator and task name, and include it in the output as `skipped` with a note that manual cleanup may be required.
3. THE Destroy_Command SHALL NOT attempt to delete resources created dynamically inside notebook code (e.g. Bedrock Agents created by `SageMakerNotebookOperator` tasks), as these are not statically discoverable from the workflow YAML.

---

### Requirement 12: Workflow-Created Resource Deletion via Operator Registry

**User Story:** As a DevOps engineer, I want Glue jobs and other named resources created by workflow tasks to be deleted as part of destroy, so that I don't have to manually clean them up after running the command.

#### Acceptance Criteria

1. THE Destroy_Command SHALL maintain an Operator_Registry mapping operator class names to deletion handlers. The initial registry SHALL contain: `airflow.providers.amazon.aws.operators.glue.GlueJobOperator` → delete Glue job using the `job_name` field via `boto3.client('glue').delete_job()`.
2. FOR EACH workflow declared in `content.workflows`, THE Destroy_Command SHALL locate the corresponding Workflow_YAML file from S3 (resolved via the stage's `default.s3_shared` connection and the `targetDirectory` of the workflows storage entry), parse all tasks, and for each task whose operator is in the Operator_Registry, extract the resource name from the designated field.
3. WHEN a Workflow_YAML file cannot be found or parsed, THE Destroy_Command SHALL log a warning and continue without treating the absence as a fatal error.
4. WHEN a Workflow_Created_Resource is not found at deletion time (already deleted or never created), THE Destroy_Command SHALL log a warning and continue without treating the absence as a fatal error.
5. WHEN a `job_name` field in a `GlueJobOperator` task contains unresolved template variables (e.g. `{proj.name}`), THE Destroy_Command SHALL log a warning that the resource name could not be statically determined and skip that task.
6. Workflow_Created_Resources SHALL be deleted before the Airflow_Workflow itself is deleted and before S3 objects are deleted, so that the workflow definition is still available for parsing during the destroy run.
7. THE Destroy_Command SHALL include all discovered Workflow_Created_Resources in the destruction plan printed during the validation phase (Requirement 3.4), so the user can review them before confirming.
